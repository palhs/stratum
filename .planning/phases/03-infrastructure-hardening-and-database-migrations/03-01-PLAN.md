Create `db/migrations/V6__reports.sql` with the following schema:

```sql
CREATE TABLE reports (
    report_id           BIGSERIAL        PRIMARY KEY,
    asset_id            VARCHAR(20)      NOT NULL,
    language            VARCHAR(5)       NOT NULL CHECK (language IN ('vi', 'en')),
    report_json         JSONB            NOT NULL,
    report_markdown     TEXT,
    data_as_of          TIMESTAMPTZ      NOT NULL,
    model_version       VARCHAR(100),
    pipeline_duration_ms INTEGER,
    generated_at        TIMESTAMPTZ      NOT NULL DEFAULT NOW(),
    ingested_at         TIMESTAMPTZ      NOT NULL DEFAULT NOW()
);
```

Design decisions (from CONTEXT.md locked decisions):
- One row per language: report_id + asset_id + language model. Each language version is a separate row.
- Keep all historical reports: every pipeline run creates a new row, no upsert/overwrite.
- Include `report_markdown` column (Claude's discretion): store pre-rendered Markdown for Phase 7 render speed, avoids on-demand rendering overhead during API reads.
- Include `data_as_of` for source data freshness (downstream phases use this for DATA WARNING sections).
- Include `model_version` (e.g., 'gemini-2.5-flash') for audit trail.
- Include `pipeline_duration_ms` for performance monitoring (Phase 9 batch validation needs this).

Add indexes:
- `idx_reports_asset_language ON reports (asset_id, language, generated_at DESC)` — primary query pattern: latest report for an asset in a specific language
- `idx_reports_asset_id ON reports (asset_id, generated_at DESC)` — query pattern: all reports for an asset across languages

Add file header comment following V1 convention:

<task type="auto">
  <name>Task 2: Create V7 report_jobs migration</name>
  <files>db/migrations/V7__report_jobs.sql</files>
  <action>
Create `db/migrations/V7__report_jobs.sql` with the following schema:

```sql
CREATE TABLE report_jobs (
    job_id      BIGSERIAL        PRIMARY KEY,
    asset_id    VARCHAR(20)      NOT NULL,
    status      VARCHAR(20)      NOT NULL CHECK (status IN ('pending', 'running', 'completed', 'failed'))
                                 DEFAULT 'pending',
    report_id   BIGINT           REFERENCES reports(report_id),
    error       TEXT,
    created_at  TIMESTAMPTZ      NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ      NOT NULL DEFAULT NOW(),
    ingested_at TIMESTAMPTZ      NOT NULL DEFAULT NOW()
);
```

Design decisions (from CONTEXT.md locked decisions):
- Simple four-state machine: pending -> running -> completed / failed. No node-level tracking in the table.
- FK relationship: `report_jobs.report_id` references `reports(report_id)`, nullable until job completes.
- Error column: TEXT for error message/traceback when status='failed'.
- One job per asset: batch runs (Phase 9) create 20 individual job rows, not a parent-child hierarchy.

Add indexes:
- `idx_report_jobs_asset_status ON report_jobs (asset_id, status)` — query pattern: find jobs for an asset by status
- `idx_report_jobs_status ON report_jobs (status, created_at DESC)` — query pattern: find all pending/running jobs

Add file header comment:
```sql

-- Phase 3 | Plan 01 | Requirement: INFRA-02
```

Do NOT modify V1-V6 migration files.
