Edit `docker-compose.yml` to add `mem_limit` to all long-running services. Use the legacy-style `mem_limit` key directly under the service definition (not under `deploy.resources` — this project uses non-Swarm deployment).

Memory limits per CONTEXT.md locked decisions:
- `postgres`: `mem_limit: 512m`
- `neo4j`: `mem_limit: 2g`
- `qdrant`: `mem_limit: 1g`
- `n8n`: `mem_limit: 512m`
- `data-sidecar`: `mem_limit: 512m`

Do NOT add mem_limit to one-shot init services (flyway, neo4j-init, qdrant-init) — they exit immediately after running and adding limits would risk migration failures on large schemas.

Place `mem_limit` directly after `image:` (or `build:` for data-sidecar) and before `restart:` for consistent formatting across all services.

Additionally, tune Neo4j JVM heap to prevent GC pause spikes:
- Change `NEO4J_server_memory_heap_initial__size` from `"512m"` to `"1G"` — setting initial = max eliminates garbage collector heap-growth pauses (Neo4j official recommendation).
- Keep `NEO4J_server_memory_heap_max__size` at `"1G"`.
- Keep `NEO4J_server_memory_pagecache_size` at `"512m"`.
- Total JVM footprint: 1G heap + 512m pagecache = 1.5GB, leaving ~500MB for OS overhead within the 2g container limit.

Add a comment above the Neo4j memory env vars:
```yaml
# JVM tuning: heap(1G) + pagecache(512m) = 1.5GB within 2g mem_limit
# initial=max prevents GC pause spikes from heap growth (Neo4j recommendation)
```
