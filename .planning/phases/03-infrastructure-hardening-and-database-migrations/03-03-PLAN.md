<objective>
Create the LangGraph checkpoint schema in PostgreSQL using a one-shot Docker init service, isolating checkpoint tables in a dedicated `langgraph` schema rather than `public`.

Purpose: The LangGraph reasoning pipeline (Phase 6+) requires PostgreSQL-backed checkpointing via `AsyncPostgresSaver`. The Python library's `.setup()` method creates tables in `public` only and has no `schema` parameter. By running raw DDL in a one-shot init service, we create the correct tables in the `langgraph` schema before the reasoning-engine ever starts. This follows the established project pattern (flyway, neo4j-init, qdrant-init are all one-shot init services).

Output: A Python init script and a Docker service definition that together create and validate the LangGraph checkpoint schema.
</objective>

Create `scripts/init-langgraph-schema.py` — a one-shot Python script that creates the LangGraph checkpoint tables in a dedicated `langgraph` schema.

Key design decisions (from CONTEXT.md locked decisions):
- Checkpoints in stratum database but in a dedicated `langgraph` schema (CREATE SCHEMA langgraph) — isolated from business tables in `public`.
- Uses psycopg3 (async psycopg) — reasoning-engine uses psycopg3, psycopg2 stays in data-sidecar.
- Does NOT use `AsyncPostgresSaver.setup()` — that method targets `public` schema only and has no schema parameter.

The script must:
1. Read `DATABASE_URL` from environment
2. Connect with `autocommit=True` (required for DDL and for checkpoint saver compatibility)
3. Execute all DDL in a single multi-statement string
4. Use `CREATE SCHEMA IF NOT EXISTS` and `CREATE TABLE IF NOT EXISTS` for idempotency
5. Print confirmation message on success
6. Exit with code 0 on success, non-zero on failure

DDL to execute (schema matches what `AsyncPostgresSaver.setup()` creates, transposed to `langgraph` schema):

```sql
CREATE SCHEMA IF NOT EXISTS langgraph;

CREATE TABLE IF NOT EXISTS langgraph.checkpoint_migrations (
    v INTEGER PRIMARY KEY
);

CREATE TABLE IF NOT EXISTS langgraph.checkpoints (
    thread_id           TEXT NOT NULL,
    checkpoint_ns       TEXT NOT NULL DEFAULT '',
    checkpoint_id       TEXT NOT NULL,
    parent_checkpoint_id TEXT,
    type                TEXT,
    checkpoint          JSONB NOT NULL,
    metadata            JSONB NOT NULL DEFAULT '{}',
    PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id)
);

CREATE TABLE IF NOT EXISTS langgraph.checkpoint_blobs (
    thread_id      TEXT   NOT NULL,
    checkpoint_ns  TEXT   NOT NULL DEFAULT '',
    channel        TEXT   NOT NULL,
    version        TEXT   NOT NULL,
    type           TEXT   NOT NULL,
    blob           BYTEA,
    PRIMARY KEY (thread_id, checkpoint_ns, channel, version)
);

CREATE TABLE IF NOT EXISTS langgraph.checkpoint_writes (
    thread_id      TEXT    NOT NULL,
    checkpoint_ns  TEXT    NOT NULL DEFAULT '',
    checkpoint_id  TEXT    NOT NULL,
    task_id        TEXT    NOT NULL,
    idx            INTEGER NOT NULL,
    channel        TEXT    NOT NULL,
    type           TEXT,
    blob           BYTEA   NOT NULL,
    PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id, task_id, idx)
);
```

Use synchronous `psycopg.Connection.connect()` (not async) for the init script — async is unnecessary for a one-shot DDL script and avoids needing asyncio. The reasoning-engine (Phase 6+) will use async.

Add a `langgraph-init` service to `docker-compose.yml` in the MIGRATION / INIT TIER section (after qdrant-init, before the INGESTION TIER comment block).

The service definition follows the established one-shot init pattern:

```yaml
  langgraph-init:
    image: python:3.12-slim
    depends_on:
      postgres:
        condition: service_healthy
      flyway:
        condition: service_completed_successfully
    volumes:
      - ./scripts/init-langgraph-schema.py:/init-langgraph-schema.py:ro
    entrypoint: >
      bash -c "
        pip install 'psycopg[binary]' --quiet &&
        python /init-langgraph-schema.py
      "
    environment:
      DATABASE_URL: postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}
    networks:
      - reasoning
    profiles: ["reasoning"]
    # No restart policy — one-shot init; creates langgraph schema for checkpoint tables
```

Key design decisions:
- `depends_on` includes both `postgres` (healthy) AND `flyway` (completed successfully) — ensures business tables exist before checkpoint schema is created, and prevents race conditions where langgraph-init connects before Flyway has finished.
- `image: python:3.12-slim` — lightweight Python image for running the psycopg3 DDL script. Consistent with reasoning-engine's future Python 3.12 base.
- `pip install 'psycopg[binary]' --quiet` — installs psycopg3 at runtime. For a one-shot init that runs once per deployment, this is acceptable (same pattern as qdrant-init which installs nothing but uses curl).
- Networks: `reasoning` only — checkpoint schema is a reasoning-pipeline concern, not ingestion.
- Profiles: `["reasoning"]` only — this init service is only needed when the reasoning profile is active. Unlike storage-tier init services, it does not need to run for ingestion-only deployments.
- No `mem_limit` — one-shot service that exits immediately after DDL.
