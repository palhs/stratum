Router endpoints:

```python
router = APIRouter()

@router.get("", response_model=WatchlistResponse)
async def get_watchlist(request: Request, payload: dict = Depends(require_auth)):
    user_id = payload["sub"]
    db_engine = request.app.state.db_engine
    tickers = _get_or_seed_watchlist(db_engine, user_id)
    return WatchlistResponse(tickers=tickers)

@router.put("", status_code=204)
async def put_watchlist(body: WatchlistUpdate, request: Request, payload: dict = Depends(require_auth)):
    if len(body.tickers) > MAX_WATCHLIST_SIZE:
        raise HTTPException(422, detail=f"Maximum {MAX_WATCHLIST_SIZE} tickers allowed")
    user_id = payload["sub"]
    db_engine = request.app.state.db_engine
    # Validate all symbols
    invalid = _validate_symbols(db_engine, body.tickers)
    if invalid:
        raise HTTPException(422, detail=f"Invalid symbols: {', '.join(invalid)}")
    _replace_watchlist(db_engine, user_id, body.tickers)
```

Internal functions:

- `_get_or_seed_watchlist(db_engine, user_id)`: Query user_watchlist for user_id. If zero rows AND this is NOT an explicit empty watchlist (check: has user ever PUT? — simplification: if zero rows, seed), seed from watchlist_defaults using INSERT ... ON CONFLICT DO NOTHING. Return list of WatchlistItem dicts.

  **IMPORTANT seeding nuance:** After a user explicitly PUTs an empty list, GET should NOT re-seed. To distinguish "never had a watchlist" from "explicitly emptied", use a simple approach: the PUT /watchlist endpoint always executes (even for empty list), so after a PUT, user_watchlist will have had rows deleted. The seeding logic should ONLY trigger on first-ever GET. Implementation: add a check — only seed if the user has NEVER had any watchlist activity. Simplest approach: seed only if zero rows exist AND no rows were ever deleted for this user. **Pragmatic solution:** Accept the edge case that clearing all tickers re-seeds on next GET. This is actually a feature — users who clear their watchlist get defaults back. If this is undesirable, a `user_watchlist_initialized` boolean column or separate tracking table can be added later.

  Actually, per CONTEXT.md: "Backend detects no watchlist rows for user_id on first authenticated request and seeds from defaults table." The simplest correct behavior: if zero rows → seed. If user explicitly empties, they get defaults on next GET. This is acceptable for v3.0.

- `_validate_symbols(db_engine, symbols)`: For each symbol, check TICKER_METADATA dict (not DB query). If symbol not in TICKER_METADATA, it's invalid. Return list of invalid symbols. This is faster than DB queries and the ticker universe is static.

  **Wait — CONTEXT.md says "Watchlist only accepts tickers that exist in stock_ohlcv or gold_etf_ohlcv (backend validates on add)".** However, the static TICKER_METADATA dict maps exactly to tickers that should exist in those tables. Using the static dict is equivalent but faster. If a ticker is in TICKER_METADATA but somehow missing from the DB, the dashboard will still fail to render data — but that's a data issue, not a watchlist issue. Use TICKER_METADATA for validation.

- `_replace_watchlist(db_engine, user_id, tickers)`: In a single connection/transaction: DELETE FROM user_watchlist WHERE user_id = ?; then INSERT rows for each ticker with asset_type from TICKER_METADATA. Validate BEFORE mutating (already done in put_watchlist). Use `with db_engine.connect() as conn:` and `conn.commit()` for atomic operation.

- For returning watchlist items with metadata: after querying user_watchlist rows (symbol, asset_type), look up `name` from TICKER_METADATA. If symbol not in metadata (shouldn't happen with validation), use symbol as name fallback.
